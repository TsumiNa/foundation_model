# rikyu replay-amount sweep — design & reproduction notes

A replay-amount sweep of the 24-task replay-based continual pre-training (a faithful reproduction of
the reference run `artifacts/continual_rehearsal_full_260524/` on the new `fm pretrain` CLI),
plus a warm-restart control that measures every task's true single-task baseline. Ran on the
RIKEN rikyu Phase 1 system (2026-06/07); all raw outputs were rsync'd to the local machine before
the Phase 1 shutdown (see the results policy below).

**Results are NOT in git** — `results/`, `reference/`, and generated `*.csv/*.png/*.html` are
git-ignored (sensitive; rsync within the team when needed). The analysis scripts skip missing
inputs gracefully, so a fresh clone runs them without errors but produces empty/partial output
until the result files are synced in.

## Design

Two run families sweep `[pretrain.replay].amount` (11 full runs, 24 tasks each,
`interval = 1`, 100 epochs/step with early stopping, 1 GB200 GPU per run):

| family | runs | meaning |
|---|---|---|
| fraction | 0.05 baseline\*, 0.10, 0.15, 0.20 | each old task replays that fraction of its own train labels per step |
| fixed count | 100, 200, 500, 1000, 1500, 2000, 2500 | each old task replays `min(n, n_train)` labels per step (clamps at 100%) |

\* the 0.05 baseline mirrors the reference config: `per_task = 0.10` on the fixed tail
(formation_energy, magnetic_moment, tc, klat, material_type); the other sweep runs are uniform.

Fractions equalize the *proportion* replayed (big tasks get thousands of labels, sparse tasks
almost none); fixed counts equalize the *absolute* number (sparse tasks get proportionally much
more, up to full). Per-task label counts for every setting: `analysis/replay_amounts.py`.

**Warm-restart control** (`jobs/warm_restart_control.sbatch`, all 24 tasks): each task alone on
its fixed full training data — `it00` = single-task pretrain, `it01..it10` = chained full-model
retrains with fresh optimizers (`fm finetune` with `freeze_encoder=false`), mimicking the
per-step optimizer reset of continual training without any other tasks. This yields two baselines
per task: the **at-intro level** (metric at the step the task was introduced *inside* the 24-task
sweep — depressed by the multi-task context) and the **single-task baseline** (best over the 11
self-restarts — the task's real ceiling when trained alone).

## Data conventions (important for reproduction)

- qc (main): `data/qc_ac_te_mp_dos_reformat_20260515.pd.parquet` — 49034 rows, columns already
  `(normalized)`, kernel-regression sequence columns + `composition`, and a **fixed `split`
  column**. No `preprocessing_path`.
- non-qc sources: `NEMAD_magnetic_20260419.parquet`, `NEMAD_superconductor_20260425.parquet`,
  `phonix-db-filtered_20260425.parquet` — raw units. The trained-on files are the derived
  `*_norm.parquet` (same stems), regenerated by `prep_nonqc_norm.py`:
  `log1p(clip(x,>=0)) -> z-score(train-split stats) -> clip(±5)`, writing `<col> (normalized)` +
  a deterministic seed-42 `split` column.
- **The `fm` CLI trains on target columns as-is** — it does NOT normalize targets
  (`[[tasks]].scaler` is inverse-only for reporting). That is why the derived files pre-normalize.
- All four files carry **file-level fixed splits**, so a task's test labels can never enter its
  training at any step (verified: zero composition overlap between qc and the other files;
  cross-file split conflicts affect ≤ ~1% of any non-qc test set).
- `material_type` is trained with the **raw 5 labels** (`num_classes=5`); the reference merged
  them 5→3, so its accuracy is not 1:1 comparable with the reference (all other tasks are).

## Configs & jobs

- `configs/full24_baseline_0p05.toml` — reference-faithful baseline (run by
  `jobs/rehearsal_full24.sbatch`).
- `configs/sweep_<tag>.toml` — the sweep variants, identical except `[pretrain.replay]`
  (run by `jobs/replay_sweep.sbatch`, config selected via the `SWEEP_CFG` env var).
- `jobs/warm_restart_control.sbatch` — the single-task control (env `TASK=<name>`).
- Outputs land in `artifacts/replay_sweep/replay_<tag>_rikyu/` and `artifacts/warm_restart/<task>/`
  (metrics at `<dir>/training/metrics_table.csv`, long format; `--resume`-capable).

## Analysis scripts (`analysis/`)

| script | output | what it shows |
|---|---|---|
| `compare_sweep.py` | stdout table | final per-task primary metric, every run vs the reference |
| `per_task_saturation.py` | `per_task_saturation.png` + `per_task_saturation/<task>.png` | per task: final metric vs labels replayed per step, with BOTH baselines (at-intro, single-task) and a `gap = g0/(1+x/k)` fit where the decay model holds (panels state why a fit is absent) |
| `replay_trajectories.py` | `replay_trajectories.png` + `replay_trajectories/<task>.png` | per task: metric after every replay event since introduction, one line per run |
| `collect_warm_restart.py` | `results/warm_restart.csv` | per-iteration single-task metrics from the artifacts mirror |
| `replay_amounts.py` | stdout table | exact labels-per-step for every task × setting |

Inputs: `results/mt_<tag>.csv` (one per run — copy of each run's `metrics_table.csv`),
`results/warm_restart.csv`, `reference/metrics_table.csv` (reference-run metrics copy).
